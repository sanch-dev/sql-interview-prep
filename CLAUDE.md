# QueryLab — SQL Interview Prep

Full-stack SQL interview coaching app targeting **Data Engineering roles**.

- **Live:** https://sanch-dev.github.io/sql-interview-prep/
- **Repo:** https://github.com/sanch-dev/sql-interview-prep

---

## Stack

| Layer | Tech |
|---|---|
| Frontend | React 18 + Vite, CodeMirror 6 |
| API | Node.js / Express (`api/index.js`) |
| Query execution | PostgreSQL (Supabase) — T-SQL translated via `api/translate.js` |
| Question schemas | Azure SQL (`querylab-server-001.database.windows.net / querylab-sql-001`) |
| Auth | Supabase |
| Build | Python 3 (`build/build.py`) |

## 9 Interview Patterns (weights sum to 100)

| Pattern | Weight | Questions |
|---|---|---|
| Window Functions | 22 | 19 questions |
| Joins & Anti-Joins | 18 | 18 questions |
| Core Aggregation | 16 | 20 questions |
| Deduplication | 12 | 11 questions |
| Date & Time Logic | 11 | 10 questions |
| Gaps & Islands | 10 | 10 questions |
| Retention & Cohorts | 7 | 8 questions |
| Filtering & NULL | 3 | 9 questions |
| Data Quality | 1 | 5 questions |

Readiness score: `Σ(pattern_weight × mastery_pct / 100)` — computed in `ProgressContext.jsx` and `patterns.js`.

Mastery levels per pattern: locked (0%) → learning (25%) → practicing (50%) → proficient (75%) → interview_ready (100%).

## Key Commands

```bash
# Rebuild questions.js + questions.json (after editing any q_*.py file)
python3 build/build.py

# Push question schemas to Azure SQL (after build)
export PATH="$HOME/.nvm/versions/node/v24.16.0/bin:$PATH"
cd api && node migrate-mssql.js

# Push schemas to PostgreSQL/Supabase (if API backend needs refresh)
cd api && node migrate.js

# Frontend dev server
cd frontend && npm run dev

# Frontend production build
cd frontend && npm run build

# Deploy: just git push — Vercel auto-deploys from main
git push
```

## Conventions

### Questions
- Fields: `id, title, difficulty, category, companies, description (HTML), schema, solution, explanation (HTML), hints (list), order_matters (bool)`
- T-SQL questions also have `"tsql": True` — build skips SQLite validation, API uses mssql dialect for reference solution
- Schema uses `VARCHAR`, `INT`, `DECIMAL` (works in both PG and Azure SQL). Avoid `TEXT`, `INTEGER`, `BOOLEAN`
- Solutions use T-SQL: `DATEDIFF`, `DATEADD`, `FORMAT`, `ISNULL`, `DATENAME`, `DATEPART`, `STRING_AGG`, `TOP`, etc.
- Build validates SQLite questions against SQLite; T-SQL questions are skipped with `SKIP` log line

### Frontend
- T-SQL only — SQLite tab removed from editor, SQLite toggle removed from Concepts page
- `EditorPane.jsx`: dialect hardcoded to `'mssql'`, comment-line button (Ctrl+/) added
- Bug-banner div stripped from Debug Lab descriptions at render time (regex in `DebugPage.jsx`)
- Expected output always visible after submission (not collapsed in `<details>`)
- Horizontal scroll on results table

### Coaching system
- `coaching.js`: keyed by question ID → `{ pattern, deContext, commonMistakes, debrief, variants }`
- `DEContextPanel` in `ProblemPane.jsx` shows pattern badge + interview frequency + DE context above question
- `CoachingBanner` is open by default
- `PatternDebrief` pops up after solving a question

### Behavior preferences
- Always commit AND push immediately after any change — no confirmation needed
- Never ask for confirmation before executing actions
- Run `migrate-mssql.js` whenever new questions are added (schemas must be on Azure SQL)
